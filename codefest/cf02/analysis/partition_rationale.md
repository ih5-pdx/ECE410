# HW/SW Partition Rationale

The kernel selected for hardware acceleration is matrix multiplication used in transformer inference, specifically the linear projection \(Y = XW\). Profiling showed that this operation dominates runtime because it performs a large number of repeated multiply-and-accumulate operations across the input activation matrix and the weight matrix. The roofline analysis supports this choice because the kernel has high arithmetic intensity compared to simpler element-wise operations, making it a strong candidate for acceleration with dedicated hardware.

The software baseline will continue to handle all non-accelerated tasks. This includes model setup, memory allocation, loading input data, loading weights, invoking the accelerator, and collecting output results. If the project remains focused on a simplified transformer-like workload, software can also manage any surrounding control flow and validation code. This keeps the hardware design focused on the single dominant kernel rather than expanding into a full model implementation.

To avoid becoming interface-bound, the accelerator must be supplied with data at a rate that matches its target throughput. In general, required interface bandwidth is approximately equal to target throughput divided by arithmetic intensity. For example, if the accelerator targets 200 GFLOP/s and the kernel arithmetic intensity is 32 FLOPs/byte, then the required bandwidth is about \(200 / 32 = 6.25\) GB/s. This provides a first-order estimate of the minimum sustained data rate needed from the host interface and memory system.

On the current hardware platform, the kernel may be limited either by compute throughput or by memory bandwidth depending on the actual processor specifications. The roofline plot makes this visible by comparing the kernel arithmetic intensity to the ridge point. A custom accelerator is expected to improve performance by dedicating hardware resources specifically to matrix multiplication and by increasing effective local data movement efficiency. Even if the kernel is partly memory-limited on the laptop CPU, the accelerator design can shift the balance by using more structured dataflow and higher internal bandwidth.
