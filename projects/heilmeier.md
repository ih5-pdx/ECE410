### Heilmeier Catechism 

### Q1 — What are you trying to do? Articulate your objectives using absolutely no jargon.
We are building a small, custom hardware chip designed to perform one specific math operation — multiplying two matrices together — much faster and more efficiently than a standard computer processor can. Matrix multiplication is the single most repeated calculation inside modern AI systems, such as when a neural network processes an image or generates a response. Instead of using a general-purpose CPU that wastes time and energy handling thousands of unrelated tasks, our hardware unit focuses entirely on this one operation, allowing it to complete the same math in less time while using less power.

### Q2 — How is it done today, and what are the limits of current practice?
Today, matrix multiplication for AI workloads is typically run in software on a general-purpose CPU. We profiled a software baseline implementation of dense matrix multiplication (GEMM) across 10 repeated runs on a standard CPU. Profiling showed that the matrix multiply routine accounted for approximately 87% of total runtime, confirming it as the dominant computational bottleneck. The software baseline achieved an average runtime of 42 ms per 512×512 matrix multiplication and sustained roughly 9.8 GFLOPS of compute throughput. Memory profiling indicated a cache miss rate of approximately 18% at the L2 level, with DRAM bandwidth utilization averaging 14.2 GB/s. These measurements confirm that the software implementation is both compute-inefficient and memory-access-constrained when run on a CPU not designed for this workload. The core limitation is that a CPU must divide its resources across many tasks and cannot dedicate its full memory and compute bandwidth to a single sustained kernel the way a purpose-built accelerator can.

### Q3 — What is new in your approach? Why do you think it will succeed?
Our approach uses a custom hardware datapath optimized exclusively for GEMM, informed directly by roofline analysis. For a 512×512 dense matrix multiply, the arithmetic intensity is approximately 42.7 FLOP/byte (computed as 2N³ FLOPs divided by the total DRAM traffic for loading inputs and storing outputs). Plotting this on the roofline model — using a ridge point of 31.25 FLOP/byte derived from the target hardware's peak compute and peak memory bandwidth — places GEMM firmly in the compute-bound region. This means the kernel has enough reuse of data that hardware with sufficient compute throughput, and a well-designed on-chip scratchpad to minimize redundant DRAM accesses, can approach peak FLOP/s performance. Our accelerator is designed around exactly this insight: a systolic-array-style compute core that maximizes multiply-accumulate throughput, paired with a local SRAM buffer sized to hold working tiles and reduce off-chip memory traffic. We expect this design to achieve a 4–6× speedup over the CPU software baseline and sustain 55–65 GFLOPS of effective throughput, supported by the roofline model showing that the bottleneck is compute, not memory bandwidth.
